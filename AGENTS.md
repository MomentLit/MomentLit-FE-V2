# AGENTS.md

## 1. 목적

이 파일은 프론트엔드 레포지토리에서 AI 에이전트와 개발자가 어떤 기준으로 작업해야 하는지 정의한다.

프론트엔드 구현은 임의 판단이 아니라, 이미 작성된 프로젝트 문서를 기준으로 진행해야 한다.

코드를 생성하거나 수정하기 전에 반드시 아래 문서를 먼저 확인한다.

```text
docs/Architecture.md
docs/design-system.md
docs/router.md
```

## 2. 반드시 참고해야 하는 문서

### Architecture.md

프론트엔드의 전체 구조를 이해하기 위한 기준 문서이다.

다음 내용을 결정할 때 반드시 참고한다.

* 폴더 구조
* 컴포넌트 분리 기준
* 페이지 구조
* API 연동 흐름
* 상태 관리 방향
* page, component, hook, service의 책임 분리

현재 문서에 없는 구조를 임의로 만들지 않는다.

필요한 규칙이 문서에 없다면, 기존 코드베이스의 구조를 우선 따른다.

---

### design-system.md

UI 구현의 기준 문서이다.

다음 내용을 결정할 때 반드시 참고한다.

* 색상
* 폰트
* 여백
* border radius
* 버튼 스타일
* input 스타일
* card 스타일
* 레이아웃 규칙
* 재사용 가능한 UI 컴포넌트 패턴

새로운 시각적 스타일을 임의로 만들지 않는다.

새 UI 패턴이 필요하다면 먼저 기존 디자인 시스템 조합으로 구현 가능한지 확인한다.

---

### router.md

라우팅 구조의 기준 문서이다.

다음 내용을 결정할 때 반드시 참고한다.

* 페이지 경로
* route 이름
* 중첩 라우트
* 페이지별 책임
* 네비게이션 흐름
* 보호 라우트 처리
* 라우트별 레이아웃

라우트를 추가, 삭제, 변경하는 경우 반드시 `router.md`도 함께 수정한다.

## 3. 개발 원칙

### 3.1 문서 우선 개발

기능을 구현하기 전에 다음 순서로 확인한다.

1. `Architecture.md`에서 구조와 책임 범위를 확인한다.
2. `design-system.md`에서 UI 규칙을 확인한다.
3. `router.md`에서 라우트와 페이지 책임을 확인한다.
4. 그 후 코드를 작성하거나 수정한다.

구현 과정에서 기존 문서와 충돌하는 부분이 생기면, 코드를 먼저 바꾸지 말고 관련 문서를 먼저 수정한다.

---

### 3.2 프로젝트 규칙을 추측하지 않는다

다음 항목은 임의로 판단하지 않는다.

* 폴더 이름
* 컴포넌트 이름
* 라우트 경로
* 디자인 토큰
* API 호출 방식
* 페이지 책임
* 상태 관리 방식

반드시 기존 문서와 코드베이스를 기준으로 판단한다.

규칙이 명확하지 않다면 기존 레포지토리에서 가장 가까운 구현 패턴을 따른다.

---

### 3.3 기존 코드 스타일을 유지한다

새 코드는 기존 프론트엔드 코드 스타일과 일관되어야 한다.

코드를 추가하기 전에 주변 파일을 확인하고 다음 패턴을 맞춘다.

* import 방식
* 파일명 규칙
* 컴포넌트 구조
* 스타일링 방식
* hook 사용 방식
* API service 사용 방식
* 에러 처리 방식
* 로딩 상태 처리 방식

## 4. 구현 규칙

### 4.1 Page 컴포넌트

Page 컴포넌트는 페이지 단위의 조립 역할에 집중한다.

Page에서 담당할 수 있는 것:

* 라우트 단위 레이아웃
* feature section 조립
* 페이지 단위 데이터 로딩
* 페이지 단위 로딩/에러/빈 상태 처리

Page 안에 재사용 가능한 UI 로직을 과도하게 넣지 않는다.

반복 사용 가능한 영역은 component로 분리한다.

---

### 4.2 Component

Component는 재사용 가능하고 책임이 명확해야 한다.

하나의 component는 하나의 명확한 역할을 가져야 한다.

Component 안에 다음 내용을 과도하게 넣지 않는다.

* 서로 관련 없는 UI 영역
* 여러 개의 비즈니스 흐름
* API 호출 로직
* 라우팅 로직

컴포넌트가 너무 커지면 작은 컴포넌트로 분리한다.

---

### 4.3 Hook

Hook은 재사용 가능한 프론트엔드 로직을 분리할 때 사용한다.

Hook에 넣을 수 있는 것:

* 데이터 요청 로직
* form 처리 로직
* modal 상태 로직
* filter 로직
* pagination 로직
* 재사용 가능한 interaction 로직

Hook 안에 JSX 중심의 UI 마크업을 넣지 않는다.

---

### 4.4 API 연동

API 호출은 UI 컴포넌트와 분리한다.

깊은 UI 컴포넌트 내부에서 `fetch` 또는 `axios`를 직접 호출하지 않는다.

프로젝트에 API/service 계층이 있다면 해당 구조를 우선 사용한다.

API 연동 시 다음 상태를 고려한다.

* 요청 함수
* 응답 처리
* 에러 처리
* 로딩 상태
* 빈 데이터 상태

---

### 4.5 상태 관리

상태 관리는 기능에 필요한 만큼만 단순하게 사용한다.

작은 UI 상태는 local state를 우선 사용한다.

여러 페이지나 멀리 떨어진 컴포넌트가 같은 상태를 공유해야 할 때만 shared state를 사용한다.

명확한 이유 없이 새로운 상태 관리 라이브러리를 추가하지 않는다.

## 5. 스타일링 규칙

모든 스타일링은 `design-system.md`를 기준으로 한다.

다음 값을 임의로 사용하지 않는다.

* 색상
* 폰트 크기
* 여백
* border radius
* shadow
* 버튼 스타일
* input 스타일

가능하면 기존 token, utility class, component style을 재사용한다.

Tailwind CSS를 사용하는 경우, 프로젝트에서 이미 사용 중인 utility 조합을 우선 따른다.

CSS Module, styled-components 등 다른 방식이 사용 중이라면 기존 레포지토리의 방식을 따른다.

## 6. 라우팅 규칙

라우트 관련 작업은 반드시 `router.md`를 기준으로 한다.

새 페이지를 만들 때는 다음 순서로 진행한다.

1. `router.md`에서 해당 route가 존재하는지 확인한다.
2. route의 책임에 맞게 page를 생성한다.
3. router에 page를 연결한다.
4. route 구조가 바뀌었다면 `router.md`를 수정한다.

임시 route나 중복 route는 만들지 않는다.

개발용으로 반드시 필요하다면 명확하게 주석으로 표시한다.

## 7. 파일 및 네이밍 규칙

파일명과 네이밍은 기존 코드베이스의 규칙을 따른다.

기본 권장 규칙은 다음과 같다.

* Page 파일: `PageNamePage`
* Component 파일: `ComponentName`
* Hook 파일: `useFeatureName`
* API 파일: `featureApi` 또는 기존 convention
* Type 파일: `feature.type` 또는 기존 convention

같은 디렉토리 안에서 서로 다른 네이밍 스타일을 섞지 않는다.

## 8. 코드 품질 규칙

생성되는 코드는 별도 요청이 없는 한 실제 서비스에 들어갈 수 있는 수준으로 작성한다.

피해야 할 것:

* 사용하지 않는 import
* 사용하지 않는 변수
* 중복 컴포넌트
* production page 안에 하드코딩된 mock data
* design system을 벗어난 inline style
* 과도한 주석
* 요청 범위를 벗어난 리팩토링
* 관련 없는 파일 수정

권장하는 것:

* 작은 단위의 컴포넌트
* 명확한 이름
* 책임 분리
* 예측 가능한 데이터 흐름
* 단순한 에러 처리
* 일관된 포맷팅

## 9. Mock Data 규칙

백엔드 API가 준비되지 않은 경우에만 mock data를 사용할 수 있다.

Mock data를 사용할 때는 다음 기준을 지킨다.

* 실제 API 로직과 분리한다.
* 임시 데이터임을 명확히 표시한다.
* 이후 API 데이터로 쉽게 교체할 수 있게 만든다.
* UI 컴포넌트 깊은 곳에 mock data를 직접 하드코딩하지 않는다.

API가 준비되면 mock data는 실제 API 연동으로 교체한다.

## 10. API 연동 작업 흐름

페이지를 API와 연결할 때는 다음 순서로 진행한다.

1. `router.md`에서 페이지의 책임을 확인한다.
2. `Architecture.md`에서 API 연동 위치를 확인한다.
3. 기존 API/service 패턴을 확인한다.
4. API 요청 함수를 작성한다.
5. page 또는 hook에서 API 데이터를 연결한다.
6. loading, error, empty state를 처리한다.
7. mock data가 있다면 제거하거나 분리한다.

## 11. 문서 업데이트 규칙

다음 변경이 발생하면 관련 문서를 반드시 수정한다.

### Architecture.md를 수정해야 하는 경우

* 폴더 구조 변경
* 새로운 계층 추가
* API 연동 방식 변경
* 상태 관리 방식 변경
* 컴포넌트 책임 기준 변경

### design-system.md를 수정해야 하는 경우

* 새로운 공통 UI 패턴 추가
* 색상/폰트/spacing 규칙 변경
* 버튼, input, card 등 공통 컴포넌트 스타일 변경
* 기존 디자인 규칙 확장

### router.md를 수정해야 하는 경우

* route 추가
* route 삭제
* route path 변경
* page 책임 변경
* navigation flow 변경
* protected route 정책 변경

문서와 코드가 다르면 문서를 신뢰할 수 없게 되므로, 구조 변경 시 문서도 함께 관리한다.

## 12. AI Agent 작업 방식

AI Agent는 바로 코드를 작성하지 않는다.

먼저 다음 내용을 확인하고 작업 계획을 세운다.

1. 어떤 문서가 관련되어 있는지
2. 어떤 파일을 수정해야 하는지
3. 기존 코드에서 따라야 할 패턴이 무엇인지
4. 문서 수정이 필요한지
5. 구현 후 확인해야 할 항목이 무엇인지

작업 요청을 받았을 때 가능한 응답 흐름은 다음과 같다.

```text
1. 관련 문서 확인
2. 기존 코드 구조 확인
3. 구현 계획 작성
4. 코드 수정
5. 문서 수정 여부 확인
6. 최종 변경 요약
```

## 13. 금지 사항

다음 작업은 하지 않는다.

* 문서를 확인하지 않고 라우트 추가
* design-system.md에 없는 스타일을 임의로 추가
* 기존 구조와 다른 폴더 구조 생성
* 하나의 컴포넌트에 과도한 책임 부여
* API 호출 로직을 여러 컴포넌트에 중복 작성
* mock data를 production 코드에 그대로 방치
* 요청받지 않은 대규모 리팩토링
* 관련 없는 파일 수정
* 문서와 코드의 불일치 방치

## 14. 작업 완료 전 체크리스트

작업이 끝나기 전에 다음 항목을 확인한다.

```text
[ ] Architecture.md 기준을 따랐는가?
[ ] design-system.md 기준을 따랐는가?
[ ] router.md 기준을 따랐는가?
[ ] 기존 코드 스타일과 일관되는가?
[ ] 불필요한 import나 변수가 없는가?
[ ] loading/error/empty state가 필요한 경우 처리했는가?
[ ] mock data가 production 코드에 섞이지 않았는가?
[ ] route 변경이 있다면 router.md를 수정했는가?
[ ] UI 규칙 변경이 있다면 design-system.md를 수정했는가?
[ ] 구조 변경이 있다면 Architecture.md를 수정했는가?
```

## 15. 기본 원칙 요약

이 레포지토리의 프론트엔드 작업은 다음 원칙을 따른다.

```text
문서를 먼저 확인한다.
기존 구조를 따른다.
디자인 시스템을 지킨다.
라우터 문서를 기준으로 페이지를 만든다.
API 로직은 UI와 분리한다.
컴포넌트는 작고 명확하게 유지한다.
구조가 바뀌면 문서도 함께 바꾼다.
```
